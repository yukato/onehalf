# データアーキテクチャ戦略

## 概要

スプレッドシート運用からの脱却と、AIレバレッジを効かせるためのデータ基盤構築戦略。

---

## データレイヤーの考え方

データは「役割」で分けて考える。

| Layer             | 役割         | 特徴                         | 技術選択                         | 例                           |
| ----------------- | ------------ | ---------------------------- | -------------------------------- | ---------------------------- |
| **Truth Layer**   | 事実の記録   | ACID、制約、監査、正規化     | RDB (PostgreSQL等)               | ユーザー、契約、決済、権限   |
| **Context Layer** | 文脈・履歴   | 柔軟、非正規化OK、追記メイン | NoSQL / JSON / Event             | ログ、イベント、チャット履歴 |
| **Meaning Layer** | 意味・類似性 | ベクトル、推論、検索         | Vector DB (Pinecone, pgvector等) | 検索、レコメンド、RAG        |

### ポイント

- 同じデータが複数レイヤーに存在してOK（例: 注文データはTruthにあるが、検索用にMeaningにもコピー）
- レイヤー間は**非同期で連携**が基本（Truthが更新 → Context/Meaningに伝播）

---

## 移行の3ステップ

```
[1. 構造化] → [2. 接続] → [3. AI活用]
   土台         配管        レバレッジ
```

### なぜこの順序か

- 構造化されていないデータをAIに食わせても精度が出ない
- 接続する前に構造化しないと、ゴミデータが伝播するだけ
- 逆順（AI先行）だと「AIで何とかする」という幻想に陥りがち

---

## Step 1: 構造化フェーズ

### 目的

スプレッドシート運用から、何かしらの管理システム（簡易なもの or 既存SaaS）に移行する。

### 重要な考え方

> システムそのものを作ることが目的ではない。
> 疎結合でいいが、それぞれがデータ的に構造化・整理されていることが大事。

### やること / やらないこと

| やること             | やらないこと       |
| -------------------- | ------------------ |
| データ構造の定義     | 完璧なUIを作り込む |
| 制約・バリデーション | 全機能を網羅する   |
| ID体系の統一         | 既存SaaSの再発明   |

### 判断基準

- 既存SaaSでカバーできる → **SaaS使う**
- SaaSにないがCRUD程度 → **簡易ツール作る**
- 複雑なドメインロジック → **ちゃんと設計する**

### 移行でAIが使えるところ

- スプシの列名・データ型からRDBスキーマ自動生成
- データクレンジング（表記ゆれ統一など）
- リレーション推定（「この列は別シートのIDっぽい」）
- 移行スクリプト生成

---

## Step 2: 接続フェーズ

### 目的

構造化されたシステム同士をつなぐ。

### 前提条件

データの構造化がされていれば、それぞれをつなぐことはやりやすい。
どのように参照すればいいかがはっきりしているのでシステムが組みやすい。

```
[イベント発生]
    ↓
[どのシステムの、どのエンティティを、どう参照するか]
    ↓
[構造化されたレスポンス]
```

### 機能するための条件

- 各システムにAPIがある（または作れる）
- ID体系が統一されている（またはマッピングがある）
- スキーマが明確（何が返ってくるか分かる）

---

## Step 3: AI活用フェーズ

### 目的

つないだものを、運用にインストールする。

### 考え方

AIは:

- 新しい仕組みを作るものではなく
- 既存の構造化された業務フローに「載せる」もの

### AIエージェント構築の技術選択

| 選択肢                        | 評価       | 備考                                    |
| ----------------------------- | ---------- | --------------------------------------- |
| Claude/GPT ネイティブTool Use | ◎ おすすめ | シンプル、保守しやすい、デバッグ容易    |
| LangChain                     | △          | 抽象化過多、デバッグ困難、API変更激しい |
| LangGraph                     | ○          | 複雑なマルチステップ処理向け            |
| Vercel AI SDK                 | ○          | Next.js連携向け                         |

**基本方針**: APIはシンプルに、疎結合に作る。

```
[FastAPI エンドポイント]
    ↓
[Claude/GPT API] ← シンプルにtool_useで呼ぶだけ
    ↓
[ツール関数] ← 普通のPython関数として独立
```

---

## 移行期間の設計

並行稼働期間の混乱を最小化するために、事前にルールを決めておく。

```
現状:   スプシA → 手動 → スプシB → 手動 → 作業

移行中: スプシA → 新システム → スプシB（並行稼働）
        ↑ ここが一番カオスになりがち

完了後: システムA → API → システムB → 自動 → 作業
```
